# 这次重构做了什么

原本所有东西都塞在一个 580 行的 `main.py` 里：配置、状态、Bark、DeepSeek（写着写着变成了在调一个根本没定义的 `GROQ_API_KEY`，这是个 bug，之前应该没真的成功跑过 DeepSeek）、人设、主动消息排程、HTML 前端全部混在一起。现在拆成这样：

```
main.py                  ← Railway 还是靠这个文件启动，内容变成几行，只负责把 app 拉起来
app/
  config.py               ← 所有环境变量在这一个文件
  state.py                ← 状态管理，把原本一堆全局变量收进一个 State 对象
  persona.py               ← Lin 的人设 + 输出格式规则（原样保留，只是去掉了重复贴了两次的部分）
  llm/
    deepseek_client.py      ← 只负责"怎么调 DeepSeek"
  agent/
    brain.py                 ← 把人设+记忆+模型串起来，生出一句回复
    proactive.py              ← 主动消息的判断逻辑（下面细讲）
  notify/
    bark.py                   ← 推送
  web/
    routes.py                  ← /watch /logs /memory /note /settings /health 这些接口
    frontend.py                 ← 原本的HTML监控台，原封不动搬过来
```

每个文件只做一件事，之后要加东西，通常只需要改一两个文件，不会牵一发动全身。

## DeepSeek 那部分改了什么

- 之前 `call_groq_api` 里用的 `GROQ_URL`、`GROQ_API_KEY` 根本没定义，等于这个功能本来就没法真的跑通。现在换成 `app/llm/deepseek_client.py`，直接调 DeepSeek。
- 模型名从 `deepseek-reasoner` 改成 `deepseek-v4-flash`。DeepSeek 在 2026/07/24（5天后）会把 `deepseek-chat`、`deepseek-reasoner` 这两个旧名字下线，所以直接用新名字，省得再改一次。你的人设里要的"先输出内心想法再输出台词"这个格式，本来就是靠 prompt 要求模型这样写，不依赖模型自己的"思考模式"，所以换成 flash 不影响效果，速度还更快、更便宜。如果哪天想让 Lin 的思考更深，把 `app/config.py` 里的 `DEEPSEEK_MODEL` 改成 `deepseek-v4-pro` 就行。
- API key 现在从环境变量 `DEEPSEEK_API_KEY` 读，你自己在 Railway 后台填。

## 主动消息（proactive）改了什么

原本是 APScheduler 每 2 小时跑一次，命中固定的几个时间段（早上/中午/傍晚/深夜）就按机率发一条。问题是：如果你们刚好聊完天，它有可能马上又在下一个时间段命中发一条，不够"看情况"。

现在的逻辑，是学 `proactive-nudge` 这个项目的核心想法（不是抄它的代码，它是给 Node 聊天网关设计的，跟这里的结构不一样，只是借了思路）：

- 后台改成每 5 分钟巡检一次（很轻量，只是查个时间差，不会真的每次都调用模型）。
- 只有"距离上次真正互动"超过设定的静默时间（默认 90 分钟），才会考虑开口。
- 到点之后，不是直接写死一句话发出去，而是丢一段"该找Anna说话了"的情境给 `generate_reply()`，跟平常聊天回复走同一条产线，这样 Lin 会记得上下文、带着记忆库的内容说话，而不是像闹钟一样喊一句固定台词。
- 发送成功后，这次"主动开口"本身也会重新计时，避免你没回应的情况下隔几分钟又被戳一次。
- 静默超过 4 小时（`max_minutes`），不管在不在预设时间段内，都会强制触发一次，这样你要是哪天作息很乱，也不会完全被晾着。

这些数字（静默多久开口、巡检多频繁）都在 `app/config.py` 顶部，之后也可以透过新加的 `/settings` 接口从前端或 Flutter 改，不用改代码重新部署。

## 顺手修的两个小 bug

- Bark 推送原本是把消息直接拼进网址发 GET，遇到中文标点容易出错；改成 Bark 官方建议的 POST + JSON，比较稳。
- `/watch` 接口的 `app_name` 字段，用新版 Pydantic 测试的时候发现如果显式传 `null` 会报错（原本代码没测出这个问题是因为你自己的前端从来没传过 null）。现在改成 `Optional[str]`，正常了。

## 部署到 Railway 需要做的事

1. 把这个压缩包解压出来的东西，整个覆盖到你本地那个 git 仓库（`anna0708120-a11y/fastapi`）文件夹里，`.git` 那个隐藏文件夹不要动。
2. 在 Railway 项目的 Variables 页面加两个环境变量：`DEEPSEEK_API_KEY`（你的 DeepSeek key）、`BARK_KEY`（原本写死在代码里那个 `qkgfpYn5LUi7pCokpYDTKi`，现在要挪到这里）。
3. 本地在项目文件夹里跑 `git add -A`、`git commit -m "重构成模块化结构，接入DeepSeek"`、`git push`，Railway 会自动重新部署。
4. 部署完打开网址，先看监控台能不能正常显示，再去对话页面发一句话试试。

## 以后要加东西，大概去哪个文件

- 加 Supabase（让状态重启不丢）：改 `app/state.py`，把每个方法的存取实现换成读写 Supabase，其他文件不用动。
- 加情绪状态：`app/state.py` 加个字段，`app/persona.py` 的 `build_system_prompt` 多接一个参数传进去。
- 接 Flutter：`app/web/routes.py` 现有的这几个接口就是 Flutter 能直接打的 API，不用另外写一套。
- 加更细的监控：`app/web/routes.py` 加新接口，或者在 `app/state.py` 加新的记录方法。
